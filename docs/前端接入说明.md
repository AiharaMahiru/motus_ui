# 前端接入说明

## 1. 目标

本说明面向 WebUI、Tauri 前端或其他调用后端 API 的客户端。

本轮后端新增了三类能力：

- 会话治理：guardrails、sandbox、MCP 暴露策略、HITL/approval 配置
- 模型能力：provider、cache_policy、声明式 response_format
- 运行时策略：memory 配置、workflow timeout/retry

前端接入时，不建议把这些配置全部堆到主输入区。推荐做成：

- 常用区：`provider`、`model_name`、`thinking`、`max_steps`、`timeout_seconds`
- 高级区：`cache_policy`、`sandbox`、`guardrails`、`MCP`、`multi_agent`、`response_format`、`memory`

## 2. 先做能力探测

前端启动后先请求：

```http
GET /api/meta
```

重点字段：

- `backend_mode`
- `supports_interrupts`
- `supports_dynamic_session_config`
- `supports_preview`
- `supports_structured_response_format`

### 推荐前端逻辑

- `supports_interrupts=false`
  - 隐藏或禁用 `human_in_the_loop`、`approval_tool_names`、中断恢复 UI
- `supports_dynamic_session_config=false`
  - 会话创建后，右侧配置面板改为只读
- `supports_preview=false`
  - 隐藏代码预览、终端预览、canvas 运行入口
- `supports_structured_response_format=false`
  - 隐藏结构化输出编辑器

## 3. Session 接口新增字段

创建和更新会话继续使用：

```http
POST /api/sessions
PATCH /api/sessions/{session_id}
GET /api/sessions/{session_id}
```

本轮新增字段如下。

### 基础模型层

- `provider`: `openai | anthropic | gemini | openrouter`
- `cache_policy`: `none | static | auto | auto_1h`
- `response_format`

### 运行时治理层

- `sandbox`
- `human_in_the_loop`
- `approval_tool_names`
- `input_guardrails`
- `output_guardrails`
- `tool_guardrails`
- `mcp_servers[].prefix`
- `mcp_servers[].allowlist`
- `mcp_servers[].blocklist`
- `mcp_servers[].method_aliases`
- `mcp_servers[].image`
- `mcp_servers[].sandbox`

### Memory 层

- `memory.type`: `basic | compact`
- `memory.compact_model_name`
- `memory.safety_ratio`
- `memory.token_threshold`
- `memory.max_tool_result_tokens`
- `memory.tool_result_truncation_suffix`

## 4. Workflow 接口新增字段

继续使用：

```http
POST /api/workflows/plans
POST /api/workflows/runs
POST /api/workflows/agent-runs
GET /api/workflows/runs/{run_id}
```

`WorkflowRunRequest` 新增：

- `runtime.timeout_seconds`
- `runtime.max_retries`
- `runtime.retry_delay_seconds`

`WorkflowRunSummary / Detail` 新增：

- `runtime`
- `attempt_count`

前端可据此展示：

- 当前运行策略
- 已重试次数
- 是否因超时失败

## 5. 推荐控件映射

### 会话创建弹窗

推荐最少展示：

- `title`
- `provider`
- `model_name`
- `thinking.enabled / effort / verbosity`
- `max_steps`
- `timeout_seconds`

高级折叠区展示：

- `cache_policy`
- `enabled_tools`
- `sandbox`
- `mcp_servers`
- `multi_agent`
- `human_in_the_loop`
- `approval_tool_names`
- `response_format`
- `memory`
- `input_guardrails / output_guardrails / tool_guardrails`

### 会话详情页

推荐把下面字段视为“当前 session 的真实生效配置”，不要再使用本地草稿态：

- `provider`
- `model_name`
- `cache_policy`
- `thinking`
- `max_steps`
- `timeout_seconds`
- `enabled_tools`
- `sandbox`
- `mcp_servers`
- `multi_agent`
- `response_format`
- `memory`

## 6. 字段编辑建议

### sandbox

推荐做成单选卡：

- `local`
- `docker`
- `cloud`

附加字段按 provider 动态显示：

- `local`: `cwd`
- `docker`: `image`、`mount_project_root`、`mount_path`
- `cloud`: `cloud_url`、`token_env_var`

### guardrails

推荐做成规则列表编辑器，每条规则至少包含：

- `kind`
- `message`
- `pattern`
- `replacement`
- `max_length`
- `ignore_case / multiline / dotall`

`tool_guardrails` 还要支持：

- `tool_name`
- `path_fields`
- `require_absolute_paths`
- `allowed_roots`

### response_format

推荐做成结构化字段表单：

- `name`
- `description`
- `fields[]`

每个字段包含：

- `name`
- `type`
- `description`
- `required`

### memory

推荐做成简单模式切换：

- `basic`
- `compact`

当选择 `compact` 时再显示高级参数。

## 7. 请求示例

### 创建支持结构化输出的会话

```json
{
  "title": "新闻摘要结构化输出",
  "provider": "openai",
  "model_name": "gpt-5.4",
  "cache_policy": "static",
  "max_steps": 256,
  "timeout_seconds": 120,
  "thinking": {
    "enabled": true,
    "effort": "medium",
    "verbosity": "medium",
    "budget_tokens": null
  },
  "enabled_tools": ["read_file", "grep_search", "to_do"],
  "response_format": {
    "name": "NewsSummary",
    "description": "新闻摘要结构化结果",
    "fields": [
      {"name": "summary", "type": "string", "required": true},
      {"name": "keywords", "type": "string[]", "required": true},
      {"name": "confidence", "type": "number", "required": false}
    ]
  },
  "memory": {
    "type": "compact",
    "compact_model_name": "gpt-5.4 mini",
    "safety_ratio": 0.5,
    "token_threshold": 256000,
    "max_tool_result_tokens": 50000,
    "tool_result_truncation_suffix": "\n\n... [content truncated due to length]"
  }
}
```

### 创建带路径限制的本地会话

```json
{
  "title": "受限文件编辑",
  "provider": "openai",
  "model_name": "gpt-5.4",
  "enabled_tools": ["read_file", "write_file", "edit_file"],
  "sandbox": {
    "provider": "local",
    "cwd": "/opt/Agent",
    "restrict_file_tools_to_workspace": true
  },
  "tool_guardrails": [
    {
      "tool_name": "write_file",
      "path_fields": ["path"],
      "require_absolute_paths": true,
      "allowed_roots": ["/opt/Agent"],
      "input_rules": [],
      "output_rules": []
    }
  ]
}
```

### 启动带超时和重试的 workflow

```json
{
  "workflow_name": "text_insights",
  "input_payload": {
    "text": "需要分析的文本"
  },
  "runtime": {
    "timeout_seconds": 30,
    "max_retries": 1,
    "retry_delay_seconds": 0.2
  }
}
```

## 8. 前端展示建议

### Session 列表

建议直接显示：

- `title`
- `model_name`
- `status`
- `total_usage`
- `total_cost_usd`

可选 tooltip：

- `provider`
- `cache_policy`
- `memory.type`

### Session 高级配置面板

优先按模块分组：

1. 模型
2. 工具
3. MCP
4. 多代理
5. Guardrails
6. Memory
7. Sandbox

### Workflow 面板

运行配置建议固定展示：

- `timeout_seconds`
- `max_retries`
- `retry_delay_seconds`
- `attempt_count`

## 9. 当前已知后端约束

- 本地 in-process backend 不支持真正的 `interrupt/resume`
  - 如果前端允许用户打开 `human_in_the_loop`，本地后端会直接拒绝创建
- HITL backend 当前仍不支持完整的动态 session 重配置
- 结构化输出能力已接入后端，但前端仍应把 assistant 消息按普通消息做兼容渲染，不要假设一定返回独立 JSON 面板

## 10. 推荐接入顺序

1. 先接 `/api/meta` 能力探测
2. 再补 session 新字段的读写
3. 再补 workflow runtime 配置
4. 最后接复杂编辑器：guardrails、MCP、response_format、memory

这样可以先让前端跑通，再逐步加高级能力，不会一次把配置面板做得过于臃肿。
