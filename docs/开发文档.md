# 开发文档

## 项目目标

本项目基于 Motus SDK 构建一个可持续扩展的智能体服务，当前重点能力包括：

- 多轮会话管理
- 工具调用
- 思考参数配置
- 用量与成本统计
- SSE 流式中间状态输出

本文件用于约束后续开发方式，确保代码质量、可维护性和可读性稳定。

## 目录说明

- `apps/`
  启动入口：
  - `apps/server.py`：HTTP 服务启动
  - `apps/tui.py`：TUI 启动
  - `apps/hitl.py`：HITL server 启动
- `core/`
  项目核心逻辑，现已按职责拆分：
  - `core/config/`：环境变量与基础配置
  - `core/llm/`：模型客户端与成本计算
  - `core/memory/`：memory 初始化与恢复
  - `core/agents/`：agent 构造与 HITL agent
  - `core/schemas/`：共享会话模型与请求响应模型
  - `core/services/`：本地会话服务
- `core/backends/`：统一 backend 协议与适配器
- `core/servers/`：HTTP 服务与 HITL server
- `core/ui/`：TUI 界面
- `core/workflows/`：workflow 注册表与实现
- `tools/`
  工具定义与技能相关代码。
- `web/`
  Web 工作台前端：
  - `src/app/`：应用壳、路由、Provider
  - `src/features/`：按功能划分的会话、聊天、workflow、tracing、runtime、meta
  - `src/shared/`：共享 API 契约、流式解析、格式化工具
  - `tests/e2e/`：Playwright 真实页面回归
- `docs/`
  开发文档与说明文档。
- `runtime/conversation_logs/`
  会话日志。
- `runtime/output/`
  运行过程中生成的输出文件。
- `skills/`
  项目内 skills，按官方推荐使用“短 SKILL.md + companion files”的目录组织形式。

## 环境变量规范

所有环境变量统一从项目根目录 `.env` 读取，不允许在代码中硬编码敏感信息。

当前主要变量：

- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`
- `FIRECRAWL_KEY`
- `APP_HOST`
- `APP_PORT`
- `OFFICECLI_BIN`

新增配置时，要求：

- 优先写入 `.env`
- 变量名清晰、语义明确
- 在 `README.md` 或本文件中补充说明

## 启动方式

本地启动：

```bash
uv run python -m apps.server
```

WebUI 开发：

```bash
cd web
npm install
npm run dev
```

WebUI 验证：

```bash
cd web
npm run build
npm run test
npm run e2e
```

如果后续补充测试、格式化或检查命令，应同步更新到本文档。

## 开发原则

### 1. 保持可读性优先

代码不只要求能运行，还要求后续维护者能快速理解。

要求：

- 命名准确，避免缩写滥用
- 函数职责单一
- 模块边界清晰
- 尽量减少隐式行为

### 2. 先抽象边界，再写实现

新增功能时，优先明确：

- 输入输出
- 状态归属
- 错误处理方式
- 是否影响现有接口

避免直接把逻辑堆进入口文件或单一函数。

### 3. 配置与逻辑分离

以下内容不应散落在业务代码中：

- 环境变量名
- 默认模型参数
- 价格映射
- 服务端口
- 工具开关

应尽量集中到 `core/` 的配置或辅助模块中统一管理。

### 4. UI 与后端解耦

后续接入新的 UI 平台时，要求：

- UI 只依赖 `core.backends` 提供的统一 backend 接口
- UI 只依赖 `core.schemas` 提供的共享模型
- 不允许 UI 直接耦合 `core/servers/` 内部实现

这样做的目的是保证：

- TUI、WebUI、桌面端、移动端可共享同一套后端能力
- 替换 transport 层时，不需要重写 UI 逻辑

### 5. Skill 组织规范

项目内 skill 应遵循：

- 路径形态：`skills/<skill_name>/`
- 入口文件：`SKILL.md`
- 详细资料：同目录 companion files，例如 `reference.md`

要求：

- `SKILL.md` 保持短小，只写触发条件、推荐流程、关键约束
- 长示例、复杂参数、详细操作说明移动到 companion files
- 通过 `load_skill` 按需加载 skill，不把整份技能说明塞进 system prompt

### 6. MCP 适配规范

项目内 MCP 集成要求统一走共享后端接口，不允许 UI 自己拼接 MCP 连接逻辑。

约束：

- MCP 配置模型统一放在 `core/schemas/`
- MCP session 构造统一放在 `core/agents/`
- UI 通过 `SessionCreateRequest.mcp_servers` 传入配置
- server 层不直接承载 MCP 业务逻辑，只负责 transport

当前支持两种 transport：

- `remote_http`
- `local_stdio`

### 7. Workflow 适配规范

workflow 不应直接耦合到某个 UI 或临时 endpoint 中。

要求：

- workflow 输入输出模型统一放到 `core/schemas/workflow.py`
- workflow 运行状态统一放到 `core/workflows/service.py`
- workflow 定义注册统一放到 `core/workflows/`
- API 层只负责暴露 workflow transport，不负责承载 workflow 业务逻辑

这样做的目的是：

- 未来接入新的 UI 平台时，不需要修改 workflow 实现
- workflow 可以单独测试、单独注册、单独扩展

当前补充约束：

- workflow run 状态至少要覆盖 `queued / running / completed / cancelled / terminated / error`
- workflow 控制统一走 `WorkflowService.cancel_run()` / `terminate_run()`
- 外部 workflow 统一从 `runtime/workflow_plugins/*.py` 或 `APP_WORKFLOW_MODULES` 装载
- API catalog 统一走 `/api/runtime/workflow-catalog`

### 8. Multi-Agent 适配规范

多代理能力要求统一走共享 schema 和 agent 工厂，不允许某个 UI 自己拼 supervisor / specialist。

要求：

- 多代理配置模型统一放到 `core/schemas/multi_agent.py`
- 主管 / 专家树的构造统一放到 `core/agents/`
- 会话层只消费统一配置，不自己拼接子代理工具
- `assistant.step` 这类流式事件必须带 `agent_name`
- usage / cost 必须覆盖主管和全部子代理，不能只统计根 agent
- 子代理输出裁剪统一走声明式 `output_extractor`

这样做的目的是：

- 后续接入新的 UI 平台时，不需要重写多代理编排
- 多代理下的成本统计保持可信
- 状态流和中间步骤可以稳定复用到 TUI、WebUI 或其他客户端

当前边界：

- 本地会话 / HTTP API 支持动态多代理
- HITL server 也支持按 session 动态注入多代理配置
- `stateful=False` 语义仍然严格按 Motus `fork()` 处理，不允许 UI 侧自造“伪共享记忆”

### 8.1 会话级模型客户端规范

模型网关覆盖必须统一走共享 schema，不允许前端直接拼 provider 私有参数。

要求：

- 会话级模型连接配置统一放到 `core/schemas/model_client.py`
- session 配置统一通过 `SessionCreateRequest.model_client`
- 当前 `base_url` 覆盖仅对 `openai / openrouter` 生效
- `api_key_env_var` 可对 `openai / anthropic / gemini / openrouter` 生效
- 不允许把原始 API Key 直接写进 session 配置

这样做的目的是：

- 保持前后端共享契约
- 支持多租户 / 多网关 / 私有代理路由
- 避免敏感信息进入会话日志或导出内容

### 8.2 结构化输出规范

结构化输出必须保持声明式 schema，而不是让 UI 直接上传任意 Python 类。

要求：

- 统一模型放到 `core/schemas/response_format.py`
- 支持 object / array / optional / nested object 组合
- 旧版 `string[] / number[]` 简写继续兼容
- 运行时模型生成统一走 `core/llm/response_formats.py`

### 8.3 动态工具宿主规范

工具宿主必须保持“内置 + 动态插件”并存，不允许把所有工具硬编码进某个 UI。

要求：

- 内置工具仍由 `core/agents/factory.py` 维护默认分组和标签
- 动态工具统一从 `runtime/tool_plugins/*.py` 或 `APP_TOOL_MODULES` 装载
- 插件模块导出 `register_tools(register)` 或 `TOOLS = [...]`
- API catalog 统一走 `/api/runtime/tools`

### 9. Tracing 适配规范

tracing 要求统一走共享 service 和稳定接口，不允许某个入口自己直接操作 Motus tracer。

要求：

- tracing 状态模型统一放到 `core/schemas/tracing.py`
- tracing 查询与导出统一放到 `core/services/tracing.py`
- session 级 trace 上下文统一放到 `core/tracing/session.py`
- API 层只负责暴露 transport，不直接拼 tracer 内部逻辑
- trace 默认输出目录统一收口到 `runtime/traces/`
- session 级 trace 默认输出目录统一收口到 `runtime/traces/sessions/<session_id>/`

这样做的目的是：

- 避免 trace 文件再次散落到项目根目录
- 避免不同 session 的 trace 混在同一导出目录里
- 后续接入新的 UI 平台时，可直接复用 tracing API
- 避免上层直接依赖 Motus 内部 tracer 结构

当前边界：

- Motus 内部 tracer 仍是 runtime 级
- 当前项目通过 session / workflow 上下文给 span 补 `session_id / workflow_run_id / project_root / trace_log_dir` 元数据
- 外部接口已经可以稳定按 session 和 workflow run 查询、导出 trace

## 注释规范

### 核心要求

后续编码时，**必须保证有充足注释**。  
这里的“充足”不是机械地每行都写，而是让阅读者在关键位置不需要反复猜测。

### 必须写注释的场景

- 关键业务流程
- 状态流转逻辑
- 非直观的兼容性处理
- 第三方 SDK 的特殊行为
- 成本计算、用量统计、缓存逻辑
- 流式输出、回调、异步并发控制
- 容易误用的函数、参数或返回值
- 临时性 workaround，但必须说明原因

### 注释内容要求

注释必须回答以下至少一个问题：

- 这段代码为什么存在
- 这段逻辑解决了什么问题
- 这里为什么不能直接写成更简单的方式
- 这里依赖了哪个外部约束

### 不合格注释示例

```python
# 设置变量
host = os.getenv("APP_HOST")
```

这种注释没有任何信息增量，不允许大量出现。

### 合格注释示例

```python
# 统一从 .env 读取服务监听地址，避免不同模块各自读取导致配置不一致。
host = os.getenv("APP_HOST", "0.0.0.0")
```

### 函数注释要求

以下情况应写函数级说明：

- 函数超过 15 行
- 函数涉及 IO、网络、状态机、异步流程
- 函数有非显而易见的副作用

建议内容：

- 功能说明
- 关键输入
- 关键输出
- 副作用

## 错误处理规范

要求：

- 不吞异常
- 错误信息可定位
- 面向接口返回可读错误
- 面向内部日志保留上下文

如果是用户可见错误，优先提供明确原因；  
如果是内部错误，至少保留失败阶段、相关参数和上下文信息。

## 接口开发规范

新增 API 时要求：

- 明确请求模型和响应模型
- 返回结构保持稳定
- 对流式接口定义事件类型
- 对会话类接口明确状态字段

如果接口影响前端或调用方，必须同步更新 `README.md`。

## 文件修改规范

修改现有代码时要求：

- 先理解再改
- 尽量局部修改
- 避免顺手重构无关代码
- 不保留无用旧逻辑

若发现历史兼容逻辑已无意义，应在确认影响后删除，不要持续堆积。

## 新增功能开发流程

建议顺序：

1. 明确需求边界
2. 确认影响文件
3. 设计数据结构和接口
4. 实现主流程
5. 补充注释
6. 做最小验证
7. 更新文档

## 文档维护要求

以下情况必须更新本文档或 `README.md`：

- 新增环境变量
- 新增服务入口
- 新增核心模块
- 修改主要接口
- 调整开发规范

## 后续执行约定

自本文件生效后，后续编码默认遵循以下要求：

- 注释必须充足
- 关键逻辑必须解释“为什么”
- 新增能力要同步补文档
- 配置统一走 `.env`
- 改动完成后尽量做最小可验证检查
