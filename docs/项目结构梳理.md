# 项目结构梳理

本次梳理的目标不是推倒重来，而是把“稳定入口”和“实现目录”分开，减少后续继续堆文件时的失控风险。

## 当前推荐理解方式

```text
apps/                  启动入口
core/chat/             会话服务、标题、消息存储、消息摘要
core/preview/          HTML / React / Python 预览运行时
core/visualization/    数据分析、图表策略、viz 重写
core/workflows/        workflow 注册、规划、运行
core/services/         薄入口层，仅保留稳定 facade
core/servers/          FastAPI / HITL 对外服务
core/backends/         UI 可复用 backend 适配层
core/schemas/          前后端共享 schema
web/                   WebUI
scripts/smoke/         系统级 smoke 脚本
skills/                项目内运行时 skills
runtime/               会话、追踪、预览、上传等运行产物
```

## 这次调整的原则

1. `core/services/` 不再继续承载大体量实现，改成稳定导出层。
2. 聊天、预览、可视化、workflow 各自成包，避免“平铺 + 相互引用”继续恶化。
3. 对外 API、session schema、SSE 事件契约不改，优先保证前端和 smoke 脚本不受影响。
4. 生成物和源码分离，后续版本控制应忽略 `runtime/`、`web/dist/`、`web/node_modules/` 等目录。

## 后续建议

1. `core/ui/tui_app.py` 仍然过大，如果后续彻底停用 TUI，可以归档或拆成组件化视图层。
2. `web/src/features/chat/pages/ChatPage.tsx` 仍然偏大，建议下一轮继续拆成状态管理、预览协同、时间线渲染三个子模块。
3. `README.md` 应只保留项目总览与启动方式，深入设计文档继续沉到 `docs/`。

2. `uv run python -m unittest discover -s tests -p 'test_*.py'`
3. `uv run python -m scripts.smoke.workflow_tracing`
4. `uv run python -m scripts.smoke.local_backend`
